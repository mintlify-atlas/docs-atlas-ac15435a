## Introduction

ClubConnect UK uses a PostgreSQL database hosted on Supabase. The schema is designed to manage schools, clubs, teachers, cover assignments, and communication workflows for after-school club management.

## Database Architecture

The database follows a relational model with clear separation of concerns:

- **Core Entities**: Schools, Clubs, Teachers, Person Details
- **Scheduling**: Cover Rules, Cover Occurrences
- **Assignments**: Teacher Cover Assignments
- **Communication**: Broadcasts, Messages, Documents

## Tables Overview

### Core Tables

<Expandable title="person_details">
  Stores personal information for individuals in the system. Used as a base table for teachers and potentially other roles.
  
  **Key Fields**: `id` (UUID), `first_name`, `last_name`, `email`, `contact`, `address`, `image`
  
  **Relationships**: Referenced by `teachers` table
</Expandable>

<Expandable title="schools">
  Represents educational institutions using the ClubConnect platform.
  
  **Key Fields**: `id` (UUID), `school_name`, `address`, `status` (enum: active/inactive/pending)
  
  **Relationships**: Has many `clubs`, `cover_rules`
</Expandable>

<Expandable title="clubs">
  After-school clubs or activities offered at schools.
  
  **Key Fields**: `id` (UUID), `school_id`, `club_name`, `club_code`, `description`, `members_count`, `status`
  
  **Relationships**: Belongs to `schools`, has many `cover_rules`
</Expandable>

<Expandable title="teachers">
  Teaching staff who can be assigned to cover clubs.
  
  **Key Fields**: `id` (UUID), `persons_details_id`, `primary_styles`, `secondary_styles`, `general_notes`, `is_blocked`
  
  **Relationships**: References `person_details`, has many `teacher_cover_assignments` and `teacher_documents`
</Expandable>

### Scheduling Tables

<Expandable title="cover_rules">
  Defines recurring patterns for club sessions (weekly, bi-weekly, monthly).
  
  **Key Fields**: `id` (UUID), `school_id`, `club_id`, `frequency`, `day_of_week`, `start_time`, `end_time`, `status`
  
  **Relationships**: Belongs to `schools` and `clubs`, has many `cover_occurrences`
</Expandable>

<Expandable title="cover_occurrences">
  Individual instances of club sessions generated from cover rules.
  
  **Key Fields**: `id` (UUID), `cover_rule_id`, `meeting_date`, `actual_start`, `actual_end`, `notes`
  
  **Relationships**: Belongs to `cover_rules`, has many `teacher_cover_assignments`
</Expandable>

<Expandable title="teacher_cover_assignments">
  Links teachers to specific cover occurrences with assignment status tracking.
  
  **Key Fields**: `id` (UUID), `teacher_id`, `cover_occurrence_id`, `status`, `assigned_by`, `invited_at`, `response_at`
  
  **Relationships**: References `teachers` and `cover_occurrences`
</Expandable>

### Communication Tables

<Expandable title="broadcasts">
  Mass communication campaigns sent to multiple teachers.
  
  **Key Fields**: `id` (UUID), `subject`, `body`, `channel_used`, `recipients_count`, `status`, `sent_by_user_id`
  
  **Relationships**: May reference `documents` (template) and `teacher_cover_assignments`
</Expandable>

<Expandable title="messages">
  Individual messages sent to or received from teachers via SMS or email.
  
  **Key Fields**: `id` (UUID), `teacher_id`, `broadcast_id`, `channel`, `direction`, `subject`, `body`, `status`, `external_id`
  
  **Relationships**: References `teachers`, `broadcasts`, and `documents`
</Expandable>

<Expandable title="documents">
  File attachments and templates used in communications.
  
  **Key Fields**: `id` (BIGSERIAL), `type`, `filename`, `storage_path`, `title`, `description`, `status`
  
  **Relationships**: Referenced by `teachers`, `broadcasts`, and `messages` through junction tables
</Expandable>

<Expandable title="teacher_documents">
  Junction table linking teachers to their associated documents.
  
  **Key Fields**: `teacher_id`, `document_id` (composite primary key)
  
  **Relationships**: References `teachers` and `documents`
</Expandable>

## Key Relationships

### School → Clubs → Cover Rules → Cover Occurrences

This is the primary scheduling hierarchy:

1. A **School** has many **Clubs**
2. Each **Club** has **Cover Rules** defining recurring patterns
3. **Cover Rules** generate **Cover Occurrences** (actual session instances)
4. **Teachers** are assigned to **Cover Occurrences** via **Teacher Cover Assignments**

### Teachers and Person Details

Teachers have a one-to-one relationship with `person_details`:

- `teachers.persons_details_id` → `person_details.id`
- This separation allows reusability of person data for other roles (e.g., admins, parents)

### Communication Flow

1. **Broadcast** is created for a campaign
2. Individual **Messages** are generated for each recipient
3. **Documents** can be attached as templates or attachments
4. External service IDs (Twilio, Resend) are tracked in `messages.external_id`

## Database Functions

### handle_updated_at()

Automatically updates the `updated_at` timestamp on record modification.

```sql
CREATE OR REPLACE FUNCTION public.handle_updated_at()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = NOW();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Applied to: `person_details`, `documents`, `schools`, `clubs`, `teachers`, `cover_rules`, `cover_occurrences`, `messages`

## Indexes

The schema includes strategic indexes for query optimization:

- **Lookup indexes**: `idx_person_details_last_name`, `idx_schools_status`
- **Foreign key indexes**: All foreign key relationships are indexed
- **Date range queries**: `idx_cover_occurrences_date` for calendar views
- **Status filtering**: `idx_messages_status`, `idx_clubs_status`
